# Cabe P6-M1 设计:WAL 提交组(Group Commit 机制核心)

> 本里程碑把 `Wal` 同步档(级别 1/3)的"每帧一写一刷"改造为 leader/follower 提交组:
> 多个并发写者的帧由轮值 leader 合并成批,一批一次 fdatasync。公开 API 单线程约束不变,
> 并发正确性由多线程并发测试驱动直打 `Wal` 验证;盘上格式与恢复链零改动。
>
> **本文为详细设计**。九个议题(静态结构 / 当选交接 / 在位循环 / 窗口衔接 / I/O 形态 /
> kWalFull / 等待唤醒 / 失败语义 / 测试)的全部裁决汇集于此。

## 1. 目标与范围

### 1.1 目标

1. 提交组协议实装:写者节点 + 无锁 MPSC 提交栈 + leader/follower 状态机,级别 1/3 的
   `WriteWal` 全部经由提交组提交。
2. 批量落盘:批内帧按到达序编码,多段窗口写出 + **批末一次 fdatasync**;盘面与 P5 等价,
   M6 恢复链零改动。
3. 失败语义全实装:结果映射矩阵、批内 `kWalFull` 切点、状态语义(滞留 / 推进 / 不全拒)。
4. 测试:单线程全量零退步 + 新建多线程并发测试目标(全绿含 TSAN 自查)。

## 3. 静态结构定稿

### 3.1 类型与常量

```cpp
// ---- wal.h 私有区 ----
struct WriterNode;                                   // 前向声明(定义在 wal.cpp;判例 RecoverCensus)

struct CommitGroup {                                 // 提交组共享原子聚合
    std::atomic<WriterNode*>   stack_head{nullptr};  // MPSC 栈顶——生产者 CAS 热点
    std::atomic<std::uint64_t> epoch{0};             // 批次纪元——唯一唤醒挂点(D3/D23)
    std::atomic<bool>          leader_active{false}; // 在位标志——窗口状态的"钥匙"(I1)
};
alignas(64) CommitGroup commit_;                     // 与 leader 私有窗口状态隔缓存行;组内不细分

bool        PushWriter(WriterNode* n) noexcept;      // 入栈;返回"由空变非空"(当选判据原料)
WriterNode* DrainAll() noexcept;                     // 取批(仅在位者);LIFO 原始链
int32_t     WaitResult(WriterNode& n) noexcept;      // 统一出口(D21/D23)

// ---- wal.cpp 匿名命名空间 ----
inline constexpr int32_t kWalResultPending = 1;      // 结果哨兵(正数;错误码全体 ≤ 0,零冲突)
                                                     // 不进 error_code.h——永不逃出 WriteWal

struct WriterNode {
    WalEntry              entry;                     // 原料值拷贝(~40B;key 字节仍在调用者内存)
    WriterNode*           next = nullptr;            // 栈链;可见性由栈顶 release/acquire 捎带
    std::atomic<int32_t>  result{kWalResultPending}; // 结果槽兼完成标志;非 wait/notify 挂点
};
```

明确不做:节点不存 seq(公开契约无消费者)、不加 `alignas`(各居各线程栈,无伪共享)。

### 3.2 生命周期三期契约(D3)

```
构造 ──► push 成功 ──────────────────► 生产者观测到 result ≤ 0 ──► 返回,栈帧消亡
     私有期            共享期                              回收期
```

- **共享期**(push 成功起):生产者只许等待 + 读 `result`,**禁止返回**;leader 可读
  `entry`/`next`、写 `result`。
- **最后触碰条款**:leader 对节点的最后一次访问必须是 `result` 的 release 写;此后任何
  指令(包括唤醒)不得访问节点。唤醒信号落在 `commit_.epoch`(生命周期与 `Wal` 等长),
  封死"notify 打在已亡栈帧"的未定义行为(经典交错:leader 回填 → follower 观测到值变
  化提前醒、返回、栈帧亡 → leader 对亡对象 notify)。
- **key 有效性三段论**:① 生产者观测到结果前不返回(共享期禁返回);② leader 对
  `entry.key` 的读取全部先于该节点的 `result` 写(最后触碰的程序序);③ 故 leader 读
  key 字节时 Put 调用必然存活。∎ 论证只依赖契约条款,不依赖运行速度。
- **禁逃逸三禁令**:禁超时(等待循环无超时分支)、禁取消(cabe 无取消机制且明文不引入
  此路径)、禁异常逃逸(push 后到观测结果之间 `noexcept` 钉死)。

### 3.3 提交栈(D4)

```cpp
bool Wal::PushWriter(WriterNode* n) noexcept {
    WriterNode* old = commit_.stack_head.load(std::memory_order_relaxed);
    do {
        n->next = old;
    } while (!commit_.stack_head.compare_exchange_weak(
                 old, n, std::memory_order_seq_cst,    // 成功:发布节点 + 参与全序 S(§4.4)
                 std::memory_order_relaxed));          // 失败:只为拿新栈顶重试
    return old == nullptr;                             // "由空变非空" = 本轮责任人
}
WriterNode* Wal::DrainAll() noexcept {
    return commit_.stack_head.exchange(nullptr, std::memory_order_acquire);
}
```

**release sequence 论证**:每个 push 都是 RMW。生产者 A 的 seq_cst CAS(含 release)
开启释放序列;B、C 的 CAS 读到链上值即延续该序列;leader 的 acquire exchange 读到序列
任意位置的值,即与 A、B、C 的发布全部同步——一个 acquire 看见整条链的完整内容。性质
依赖"每环都是 RMW":任何环节改普通 store,序列即断。

**ABA 与安全回收免责**:取批是一次 exchange 全摘、无比较,无 ABA;push 侧不解引用比较
值,ABA 无害。节点归调用者栈所有、结构从不释放,hazard pointer/epoch 回收整个问题不存在。
免责来自三个上游决策(全摘 + 栈上自有 + follower 必等),**未来不得"优化"成弹单节点或
堆上节点**——那会把三类经典无锁难题一起请回来。

### 3.4 落位、命名与移动语义(D5)

| 实体 | 命名 | 位置 |
|---|---|---|
| 写者节点 | `WriterNode`(`entry`/`next`/`result`) | wal.h 前向声明 / wal.cpp 定义 |
| 共享原子聚合 | `CommitGroup` / 成员 `commit_` | wal.h 私有区 |
| 结果哨兵 | `kWalResultPending`(=1) | wal.cpp 匿名命名空间 |
| 入栈 / 取批 / 出口 | `PushWriter` / `DrainAll` / `WaitResult` | `Wal` 私有方法 |
| 链反转 | `ReverseChain` | wal.cpp 匿名命名空间(不碰 `Wal` 状态) |

- 公开接口**零暴露**:`WriteWal` 签名与阻塞语义不变,wal.h 公开区无提交组类型;测试只
  走公开口,不开观测后门("不为测试扩 API"延续)。
- **移动语义雷**:`std::atomic` 不可移动。`Wal(Wal&&)`/`operator=` 须手工 relaxed
  load/store 搬运三原子,合法前提写成契约:**移动仅发生在 Open 编排期、提交组静止时**
  (栈空、无人在位、无 waiter),移动函数内加静止断言(`stack_head == nullptr` 等)。
- **Open/Recover 后提交组零动作**:三原子的默认初值(栈空、纪元 0、无人在位)恰好是
  正确的静止态;恢复链重建窗口状态时不碰提交组。

## 4. 协议状态机与证明

### 4.1 状态机(D6)

```
 S0 构造节点(私有期)
   │ push(带回"由空变非空")
   ▼
 S1 已入栈 ──当选裁决──┬─ 任职 ─► S2L 在位 ◄──連任──┐
                      │           │ 取批→反转→批处理→ │
                      │           │ 回填+唤醒 ────────┘(复查非空)
                      └─ 跟随 ─►  │(复查为空)
                         S2F 等待  ▼
                          │      S2X 卸任序列(让位→复查→条件再任职)
                          │        │(无活 / 已有人接班)
                          ▼        ▼
                         S3 离场(统一 WaitResult(node) 读结果返回)
```

六个状态、六类事件(push / 当选裁决 / 取批 / 落盘回填 / 卸任 / 再任职)。两个建模决策:
leader 也走 S3(返回值不开特殊通道——空批场景下任何"本地捷径"都会断);S2X 单列(交接
缝隙的论证作用域)。**排除"leader 中途消亡"分支**:D3 禁异常逃逸、cabe 无线程取消、盘写
失败是正常返回错误码走回填——不设心跳/接管机制,此排除立此存照。

### 4.2 三条不变量(证明靶子)

- **I1(钥匙制·唯一性)**:在位标志任意时刻至多被一个线程持有;窗口状态机、WAL 缓冲、
  盘面、`DrainAll` 执行权随标志走——仅持有者可碰。取得 = `exchange(true)` 返回 false,
  让出 = `store(false)`;让出后的线程(S2X 后半段)只许访问共享原子。
  (注:早期表述"S2L∪S2X 至多一线程"偏强——旧 leader 让位后走 ②③ 期间,新 leader
  可以已在 S2L 取批,系统无危险。排他单位是钥匙,不是状态区间。)
- **I2(恰好一次)**:每个完成 push 的节点被恰好一次 `DrainAll` 摘走、`result` 被回填
  恰好一次。
- **I3(无遗弃·活性)**:栈非空时,有限步内必发生一次 `DrainAll`。

### 4.3 当选与卸任(D7/D8)

**当选**:`PushWriter` 返回 true(本轮栈生命周期的唯一责任人——栈只有取批能清空,故每
两次取批之间"由空变非空"恰好发生一次)者执行 `leader_active.exchange(true)`,返回 false
即任职,否则跟随。其余写者全程不碰标志(标志行 RMW 从每批 k 次降到约 1 次)。

**卸任序列(S2X,次序不可交换)**:

```cpp
commit_.leader_active.store(false, std::memory_order_seq_cst);      // ① 先让位
if (commit_.stack_head.load(std::memory_order_seq_cst) != nullptr)  // ② 再复查
    if (!commit_.leader_active.exchange(true, std::memory_order_seq_cst))  // ③ 条件再任职
        /* 回 S2L 入口(从 DrainAll 重新开始) */;
// 离场 → S3
```

- **②①交换即遗弃**:先复查(空)后让位的缝隙里,新写者可完成"push + 当选失败"全套
  而无人知晓。正序下两个操作作用于同一原子,必有先后:写者在后则读到 false 自己当选;
  写者在前则 leader 的复查必见其节点(可见性由 §4.4 的全序保证)。
- **③不抢回不得取批**:让位后可能已有新 leader 在工作,无钥匙取批即双写者(违 I1)。
  抢不回可安心离场:新任无论从哪条路上任,S2L 首动作都是 `DrainAll` **全摘**。
- **两种复查分工**:在位连任复查(S2L 内,不让位,acquire,性能口,允许漏看)vs 卸任
  复查(②,seq_cst,安全口,不许漏看)。议题边界:在位循环不得改动卸任序列。

### 4.4 内存序裁决(D9)

**缺口**:生产者"写栈→读标志"与 leader"写标志→读栈"是两条各读对方未写变量的腿,
acquire/release 无 reads-from 可挂靠(它管"读到之后",不管"能否读到"),存在双方都读
旧值 → 节点遗弃的 Dekker 型 store-load 交错(x86 唯一允许的重排恰是它,非弱序架构专属)。

**裁决**:四操作 seq_cst 进入全局单一总序 S——push 成功 CAS、当选 exchange(S1 与
S2X③ 两处)、卸任 store、卸任复查 load。配套:

- **在位标志兼任窗口状态交接棒**:`cur_off_`/`n_frames_`/`seq_next_`/缓冲皆普通变量,
  前任写、新任读;可见性 = 卸任 store 的 release 面发布 + 当选 exchange 的 acquire 面
  承接。seq_cst 自含此语义,**两操作永不得降级**。
- **x86 成本注**:lock 前缀 RMW 本即全屏障,seq_cst 零增量;seq_cst store 编译为
  `xchg`,每批一次;价值在未来弱序架构移植时正确性不丢。

### 4.5 三证明

**I1**:`leader_active` 修改序中"取得(false→true 的 RMW)"与"让出(true→false 的
store)"严格交替——任何 exchange 取钥匙必须读到 false,而两次取得之间在修改序上必有一次
让出,让出仅持有者执行;RMW 原子性排除插队。∎(单变量论证,不需要 S。)

**I2**:节点每次调用新构造、push 恰一次;`DrainAll` 的 exchange 是破坏性全摘,链整条
离栈,同一节点不可能出现于两次取批的返回链;取批与回填仅钥匙持有者执行(I1),程序结构
上批内每节点恰好回填一次。∎

**I3**:对任意 push(其所在周期责任人的当选 exchange 记 Rp,该 push 的 CAS 记 Wp;
Wp 程序序先于 Rp),设有 leader 走卸任序列(store false 记 Wl,复查 load 记 Rl;
Wl 程序序先于 Rl)。四操作均在 S 中,对 Rp 与 Wl 的 S 序分情况:

- **Wl <ₛ Rp**:责任人的 exchange 在让位之后,必读 false → 当选竞争必有人得钥匙,
  得者 `DrainAll` 全摘,节点被处理;
- **Rp <ₛ Wl**:则 Wp <ₛ Rp <ₛ Wl <ₛ Rl(程序序嵌入 S)→ leader 的 seq_cst 复查
  必见该节点(或见到更晚的取批结果——那它已被处理)→ 走 ③:再任职成功则亲自全摘;
  失败则钥匙已被 Wl 之后的他人取得,新任首动作 `DrainAll` 全摘,节点必在其中。

两情况皆终结于覆盖该节点的一次 `DrainAll`。连任复查(acquire)漏看不伤 I3:漏看只把
执行引向卸任序列,而卸任序列已在证明中兜底。∎

### 4.6 在位循环(D10/D11/D12)

```cpp
// S2L:持有钥匙进入(S1 当选或 S2X③ 再任职)
for (;;) {
    WriterNode* batch = DrainAll();                       // acquire
    if (batch != nullptr) {
        batch = ReverseChain(batch);                      // LIFO → 到达序
        ProcessBatch(batch);                              // §5~§8:编码+落盘+回填+唤醒
    }
    if (commit_.stack_head.load(std::memory_order_acquire) == nullptr) break;
    // 非空 → 连任(无限,D12)
}
// → S2X 卸任序列(§4.3,次序不得改动)
```

- **回填与唤醒先于连任复查**:每批生命周期完整闭合,最后触碰条款可论证。
- **空批容忍**:leader 自己的节点可能已被前任的连任循环处理(push 后、当选前被全摘)。
  空批跳过处理直接复查。**终值保证**:能当选 ⇒ 前任已让位 ⇒ 前任"回填全部 → 复查 →
  让位"的程序序保证本线程结果已是终值——S3 读结果不存在悬空中间态。
- **到达序定义**:各写者 push CAS 在栈修改序中的先后。反转(`ReverseChain`,标准指针
  反转,链已离开共享世界无并发)使批内 seq 序 = 到达序。反转的正确性必要性已排除
  (阻塞语义 ⇒ 同写者两帧必分属前后批;同 key 并发被 P7 路由结构排除),保留理由:
  seq 与 timestamp 大体单调的诊断直观性、对 P7 异步多帧提交的防御、O(批) 成本。
- **无限连任**:设上限 = leader 看见活却拒绝再任职,而栈内节点的责任人已当选失败在睡,
  卸任复查是它们唯一兜底——拒绝即破坏 I3,除非另造"指定继任者"协议(等待协议加唤醒
  语义、状态机加路径、三证明加分支),为 P6 不可触发的问题付出不成比例复杂度。已知局限
  记录:持续饱和下 leader 调用者延迟无上界(P6 公开 API 单线程不可触发;并发测试驱动工作量
  有限必终止;P7 事件循环集成时重评)。批的时长天然有界:批 ≤ 并发写者数(§5.4)。

## 5. 批处理管线(D1/D13/D14/D15)

### 5.1 管线选型(D1)

复用攒批档机械:帧在窗口连续打包,窗口攒满做**中段写出**(段写,无 fsync),批末做
**末段写出 + 一次 fdatasync**。对比"逐块写、批末刷"的否决理由:I/O 合并不彻底(32K
窗口 8 次 4K 写 vs 1 次 32K 写)+ 平添第三条管线;复用方案让同步档与攒批档共用一套
打包/写出/变体 Y 代码,差异收窄为 **fsync 时机 + 返回时机** 两个策略。

### 5.2 批内循环与统一准入

```cpp
// ProcessBatch(batch):仅钥匙持有者执行
for (n : batch) {                                   // 到达序
    // ---- 统一准入(一帧空间;两条进墙路径必须合并在同一函数)----
    // 路径甲:窗口本来是 0(上批留下的满态)→ 惰性 WindowAt(cur_off_);仍 0 → 撞墙
    // 路径乙:窗口攒满 → WriteOutWindow()(中段写出,内部重开窗);重开得 0 → 撞墙
    //   撞墙:n 及其后全员 kWalFull,跳收尾(§7);写出失败:停止推进,全员
    //   kWalWriteFailed(§8),跳收尾
    Append(n->entry);                               // 编码 + seq_next_++(单点分配在此)
}
末段 WriteOutWindow();                               // 可空(最后一帧恰好触发过中段写出)
若本批写出过任何字节 → dev_.Sync();                  // 批末唯一 fdatasync(失败 → §8 矩阵)
统一收尾(§9.2):逐节点回填 → 纪元递增 → notify_all
```

- **seq 单点分配**(D13):代码不动(`EncodeFrame(e, seq_next_++)` 仍在 `Append`),
  变的是执行资格——从"唯一线程"换成"I1 唯一钥匙持有者"。leader 按到达序逐帧 `Append`
  ⇒ seq 稠密递增 + 盘上按 seq 紧凑排列,M5 不变量③与 M6 走读单判据的根基原样。
- **入口检查三分**(D13):key 长度预检(纯函数)与 `cur_buf_` 装置态守卫(**装置期定
  死、运行期只读**——与移动语义同款静止期论证,生产者无同步读取的依据,条款立此)留
  生产者侧 push 之前,非法请求不进栈;窗口准入(`window_bytes_`/`WindowAt`)迁入 leader。
  对外语义微变如实记录:环满时 `kWalFull` 从入口立即返回变为经批处理回填——返回值与
  状态后果不变,Engine 救援无感知,仅时延加一次批处理。
- **三条简化性质**(D14):中段写出必整块(窗口恒为 4K 倍数 ⇒ 攒满即整块,搬运量恒 0);
  变体 Y 搬运只发生在批末段;末段可为空但 fdatasync 无条件(中段写出全部未刷)。
- **`WriteOutWindow` 重构**(D14):从 `Flush` 抽出"写出 + 变体 Y 搬运 + 重开窗"
  (无 fsync),`Flush = WriteOutWindow + Sync`——级别 2/4 对外行为零变化;"失败不动
  任何状态、内容滞留、下次原样重试"语义由 `WriteOutWindow` 继承。
  (实装注:字面组合 "`WriteOutWindow` 后接 `Sync`" 会改变 P5 `Flush` 的失败语义——
  P5 是"写出 → fdatasync → **才**推进",fsync 失败时状态不动、整段滞留重试;若先推进
  再 fsync,fsync 失败后状态已前进,滞留语义破坏。实际落地为三件套:`WriteWindowSpan`
  (仅写出,不动状态)+ `AdvanceAfterWrite`(搬运+推进+重开窗,纯内存)+ 组合——
  `WriteOutWindow = Span + Advance`(批路径,无 fsync),`Flush = Span + Sync + Advance`
  (P5 次序逐字保持)。两档的次序不对称正是 §8.2 "欠账"辩护的代码形态。)

### 5.3 盘面等价论证(D1 的两段证明)

**正常态逐字节等价**:批写出的帧位置与内容 = P5 同步档逐帧路径的产物(帧紧凑、整块
对齐、尾块"帧+补零");单线程退化(批大小 1)下逐调用等价:每调用一批、批末必 fsync、
尾块同字节续写——恢复扫描无法区分盘是谁写的。

**崩溃态落在 M6 容差内**:撕裂范围从"一个 4K 块"(P5 同步档)放宽到"一个窗口"
(= P5 攒批档形态),恰为 M6-D13 碎片双条件容差的设计域(物理一窗不跨缝 + seq 一窗
帧数)。两个补充论证:① 已确认帧不受伤——批首段同字节重写上一批尾块,撕裂落点无论
何处旧帧字节不变(M2 撕裂安全论证原样);② 未确认帧丢弃不违约——本批写者全部还挂着,
级别 1/3 的承诺尚未对其生效(与 M6-D13"只抹契约内可丢帧"同构)。

**幽灵写注记**:中段写出的帧在批末 fsync 前崩溃,若恰已被设备持久化,恢复将作为合法
活帧重放——"调用者没收到成功,数据却恢复出来"。非 P6 新语义(P5 fsync 失败即有),
统一口径见 §8.3。

### 5.4 批大小上界(D14)

**批 ≤ 并发写者数**:`WriteWal` 阻塞至结果回填,每写者任意时刻至多一个节点在栈;批 =
一次全摘 ≤ 写者总数。上界与窗口容量无关——批可超一个窗口,由中段写出消化(多段写、
仍一次 fsync)。推论:follower 等待 ≤ 两个批处理时长(在途批剩余 + 自己所在批),每批
有界于 O(并发写者数) 帧写出 + 一次 fdatasync——P6-D5"无配置项"的结论不变,上界来源
从"窗口容量"修正为"阻塞语义 + 并发度"(P6 README 同步修正)。

### 5.5 旧码处置与等价链(D15)

**删除**(调用方全部消失):`SyncCurrentBlock` 整函数;`Append` 的
`if (sync_level() && n_frames_ == kWalFramesPerBlock)` 块推进分支(`Append` 退化为纯
"编码+拷贝+计数",攒批档从未走过该分支,删除零影响);`WriteWal` 同步档旧函数体。
**保留**:`sync_level()` 分发、`WindowAt`/窗口状态/贴缝绕回/恒留一块、常量、攒批档路径。

**结构收益**:缓冲表示法两套归一(同步档"单块暂存"消失,只剩窗口);两档差异 = fsync
时机 + 返回时机;运行时切档不再有表示法转换(缓冲内留窗帧两档含义相同),Engine 侧
"收紧先 Flush"保留(语义仍需补持久化欠账)。

**单线程等价四链**(零退步依据):① 持久化语义——每调用自成一批,批末必 fsync,返回
即持久;② 盘面演进——批大小 1 的批末写出 = 尾块整块同字节重写 + 追加,与 P5 块内续写
字节相同;③ `kWalFull` 判定点——P5 块推进准入与 P6 窗口重开准入是同一段 `WindowAt`
几何;④ 成本微变注脚——批末清零缓冲已用之外部分(≤32K memset,约半微秒)替代 P5 块
推进时的 4K memset,对比毫秒级 fdatasync 可忽略。**测试原则:用例失败先疑等价链,不改
测试。**

## 6. 持久化论证(D16/D17)

### 6.1 fdatasync 覆盖范围(D16)

命题:批末一次 `dev_.Sync()` 使本批全部段写持久。两层:① O_DIRECT 写完成 ≠ 持久——
`pwrite` 返回仅表示绕过页缓存到达设备,数据可能停在设备易失写缓存(此即幽灵写的物理
来源);② 块设备 fdatasync = 内核向设备发 Flush 命令(NVMe Flush),规范语义为"命令
到达前所有已确认完成的写必须落非易失介质"——不区分来源与偏移,刷整个设备写缓存。配合
`pwrite` 同步性(上段完成才发起下段),批末一次覆盖全部。**判例**:M6 碎片抹除即"多次
`WriteAt` + 末一次 `Sync`"同款模式,P5 已验证。**边界规则:中段写出失败立即停止推进,
不许带伤继续**——继续写会主动制造"中间缺段"的矛盾盘面(恢复会正确拒开,但不应制造)。

### 6.2 工程确认五项(D17)

1. `RawDevice` 零新接口(只用 `WriteAt`/`Sync`);向量化写否决——每段在缓冲与盘上都
   连续(窗口不跨缝),普通 `pwrite` 即一段;
2. O_DIRECT 三对齐(偏移 = `cur_off_` 恒 4K 对齐;长度 = `AlignUp`;缓冲 =
   `AllocAligned`)与现有 `Flush` 同源,Debug 断言现成;
3. **批内每物理块至多写一次**:段与段在盘上严格向前推进,无写写顺序依赖(§6.1 的覆盖
   论证因此不需考虑冲突);跨批尾块同字节重写发生在上批 fsync 之后,先后由批串行性(I1)
   保证;
4. 与 io_uring 后端无关:WAL 直接持有 `RawDevice` 不走 `IoBackend`(P5M2-D6),双后端
   下提交组执行同一份代码;io_uring 档位验证的是 value 路径与改造无相互干扰;
5. 系统调用账(红线结构论证):单线程退化路径每 Put = 1 次 4K `pwrite` + 1 次
   fdatasync,与 P5 **完全相同**;新增项全为纳秒级(约 5 次无竞争原子 + memset 注脚)。
   竞争收益(信息性):k 写者一批 ≈ 段数次 `pwrite` + 1 次 fsync,对比 P5 串行 2k 次
   系统调用。

## 7. `kWalFull` 切点与救援衔接(D18/D19/D20)

### 7.1 切点:前缀成功、后缀全拒(D18)

撞墙帧干净拒绝(帧未编码、seq 未耗、可完整重试——P5 性质延续)。**批内余帧一律
`kWalFull`,不逐帧再探**:环上空间唯一释放口是回收,而回收在 P6 不与 leader 并发
(P6-D9)——批处理期间环可用空间单调不增,再探必然同 0。附带"到达序公平"条款(批内
结果恒为"前缀成功、后缀全拒"),**标注为对 P7 的协议承诺**:届时即使并发回收让空间
中途出现,余帧仍一律拒,保住"批内结果与到达序一致"。不变量检查:后缀 seq 未耗无空洞;
被拒帧无残留状态;混合结果本就是逐帧准入的设计内形态。

### 7.2 撞墙批收尾(D19)

**结构性质:撞墙时刻不存在"已编码未写出"的本批帧**。撞墙仅现于两个开窗关口——批中段
窗口重开失败(其前刚写出整窗,`n_frames_ == 0`)与批首惰性重开失败(本批尚未产出帧;
缓冲内至多是上批已持久的留窗帧)——而批末写出永远在已准入窗口内进行(**窗口内写出永不
缺空间**:空间在开窗时已付)。收尾四步:

1. 后缀停止处理;
2. 本批写出过字节才 fdatasync(批首撞墙零写出则跳过);
3. 回填:前缀的"成功"必待 fsync 返回后(§6.1 纪律);fsync 失败前缀进 §8 改判,
   **后缀仍 `kWalFull` 不株连**(从未碰盘,该码准确且保留"重试即可"语义);
4. 单次统一唤醒,不分段。

撞墙批后 `Wal` 停在 P5M5-D12 满态惰性标准姿态,救援腾空间后下次准入自然复活,零新机制。

### 7.3 Engine 救援衔接:零改动(D20)

`WriteWalRescuing`(engine.cpp:479-492)四环复核:① `kWalFull` 语义不变(仅到手时延
经批,救援无感知);② 救援在静止态执行(`WriteWal` 已返回,无 leader 在飞,P6-D9 自动
满足)——含小论证:`Flush` 对同步档的提前返回(wal.cpp:548)在 P6 仍正确,每批 fsync
收尾 ⇒ 已分配 seq 的帧皆持久,留窗帧只是盘上字节的副本,非欠账;③ 回收几何与 P5 同构
(P6 同步档盘面 = P5 攒批档盘面,P5M5 §7 边界语义逐字成立);④ 重试 = 批大小 1 = 仅需
一块,回收后余量(环 ≥ 缓冲 + 4K,Open 校验)必过——"重试恰一次必成"的注释原样保留。
**P7 记账**:并发下多 Put 同时 `kWalFull` 的救援串行化 + "恰一次"失效(空间被并发重试
者竞争)→ 救援策略重议,流向 P7(§14)。

## 8. 失败语义(D24/D25/D26)

### 8.1 结果映射矩阵(D24)

| 失败点 | 甲群(已写出) | 乙群(已编码未写出) | 丙群(未编码) |
|---|---|---|---|
| 中段写出失败 | `kWalWriteFailed` | `kWalWriteFailed` | `kWalWriteFailed` |
| 末段写出失败 | `kWalWriteFailed` | `kWalWriteFailed` | (不存在) |
| 批末 fdatasync 失败 | `kWalWriteFailed` | (不存在) | (不存在) |
| 撞墙批 fsync 失败(D19 例外行) | 前缀改判 `kWalWriteFailed` | — | 后缀维持 `kWalFull` |

原则:**回填的码描述失败根因,不描述节点个体处境**。写失败是设备故障,丙群单独给别的
码等于暗示"马上重试可能成功"——设备刚失败,误导;`kWalFull` 是资源状态,对后缀准确且
可行动。两条裁决同源。失败点越靠后存活群越少,矩阵自然退化——实现即单一失败出口:
停止推进 → 全员同码 → 统一收尾(§9.2)。

### 8.2 失败后状态语义(D25)

- **乙群滞留**:seq 已耗不可清(清除即挖洞,M6 走读判据把洞后全部判不可信)。安全前提:
  编码帧自包含(`EncodeFrame` 已拷 key 字节入帧体),帧主人离场后无悬空引用。
- **中段失败状态不动**:`WriteOutWindow` 失败时窗口/缓冲/指针全保持;下一批准入检查
  发现窗口满,触发同偏移同字节的隐式重试(= P5 攒批档"清滞留"同机制,撕裂安全论证
  原样)。设备恢复则服务自然续,未恢复则下批同败,无累积损害。
- **fsync 失败状态保持推进,不回滚**:本批写出已全部成功,字节已在设备(可能已持久);
  下一批批末 fdatasync 按全覆盖语义顺带确认。与 P5 `Flush` 原地重试的不对称辩护:P5
  攒批档帧主人在入缓冲时已被告知成功,`Wal` 欠其尽力持久化;P6 同步档帧主人拿到的是
  失败,不欠确认,缺的只是一条 Flush 命令,任何未来 Flush 都能补上。
- **不进全拒态**:P5 无毒化先例;逐项检查(seq 稠密保持 / 盘面推进单调 / 滞留重试撕裂
  安全 / 恢复对任意前缀盘面有定义)找不到"继续服务破坏不变量"的路径;毒化把可自愈的
  设备瞬态(忙 / 超时 / 精简配置池满)升级为永久拒服务。M2 对账验证此论断,反例则重议。
  Engine 不毒化(错误原样上抛,用户决定重试,P5 同构)。
- **幽灵写三来源统一口径**:① 崩溃前未确认的中段帧;② fsync 失败批的已写帧;③ 滞留
  乙群帧。口径:**凡调用者未收到成功的帧,最终命运未定义;恢复以盘上证据为唯一真相**
  (M6 总原则的推论,P6 仅新增两条通往它的路)。

### 8.3 错误码与分层(D26)

零新增:写失败 `kWalWriteFailed`(-103001)、撞墙 `kWalFull`(-103002)、超长 key
`kWalKeyTooLong`(-103000)全复用;`kWalResultPending` 是内部哨兵不进错误码表。返回值
分层约定不变:提交组全在 `Wal` 内部,对外仍 `int32_t`,Engine 的 `Status` 转换点零变化。

## 9. 等待与唤醒(D21/D22/D23)

### 9.1 等待协议(D21)

```cpp
int32_t Wal::WaitResult(WriterNode& node) noexcept {
    for (;;) {
        int32_t r = node.result.load(std::memory_order_acquire);
        if (r != kWalResultPending) return r;              // ① 查结果
        std::uint64_t e = commit_.epoch.load(std::memory_order_acquire);  // ② 拍快照
        r = node.result.load(std::memory_order_acquire);
        if (r != kWalResultPending) return r;              // ③ 再查(性能优化,非正确性必需)
        commit_.epoch.wait(e, std::memory_order_acquire);  // ④ 纪元仍为 e 才入睡
    }
}
```

**无丢失唤醒论证**(对"③读到哨兵"分两情况):④时纪元已变 → `wait` 不睡立即返回,
①必读终值(递增的 release 链保证回填先行);④真入睡 → 递增尚未生效,其后必有"递增 →
notify",notify 必在入睡之后,futex 必中。事故场景("唤醒先于入睡且其后再无唤醒")被
条件入睡封死:唤醒若真在前,递增亦在前,`wait(e)` 因值不等拒绝入睡。跨批次误醒(别的
批完成的 notify_all)= 虚假唤醒,循环消化,无害。

### 9.2 唤醒协议:统一三步收尾(D22)

```cpp
for (n : batch) n->result.store(rc_n, std::memory_order_release);  // ① 回填(最后触碰)
commit_.epoch.fetch_add(1, std::memory_order_release);             // ② 纪元递增
commit_.epoch.notify_all();                                        // ③ 广播
```

①先于②:follower 凭新纪元信任结果为终值的 release 链;②先于③:"先改值后通知"铁律
(换序存在"通知打空 + 带旧快照入睡 + 其后无通知"的卡死时序)。**统一收尾纪律:成功批 /
撞墙混合批 / 失败批全部从这同一个出口离开,无路径可绕过。** `notify_all` 采纳(`notify_one`
需链式接力,平添失败模式);惊群账:误伤 ≤ 总线程数 − 批大小,单次约一趟 futex 往返 +
两次 L1 读,淹没在 fsync 中;"每批独立等待字 + 代际环"否决(引回已消灭的生命周期难题),
P7 并发规模质变时重开。

### 9.3 出口、纪元与内存序总表(D23)

- **统一出口**:两角色皆 `return WaitResult(node)`;leader 路径前置
  `assert(node.result.load() != kWalResultPending)`(终值保证可执行化——Debug 破坏即
  断言,Release 退化为可排查的挂起而非返回垃圾)。
- **纪元 `uint64`**:libstdc++ 对 64 位原子 wait 走代理 futex 表(哈希冲突的额外虚假
  唤醒由循环消化,一层间接纳秒级);`uint32` 直接 futex 因理论回绕混淆否决——代理开销
  测不出,不为省它买理论坏路径。
- **内存序总表**(实装与审查基准):

| 操作 | 内存序 | 出处 |
|---|---|---|
| push CAS(成功 / 失败) | seq_cst / relaxed | D9 / D4 |
| 当选·再任职 exchange | seq_cst | D9 |
| 卸任 store(false) | seq_cst | D9 |
| 卸任复查 load(栈顶) | seq_cst | D9 |
| 连任复查 load(栈顶) | acquire | D9 |
| 取批 exchange | acquire | D4 |
| result 回填 store | release | D3 |
| result 读取(等待循环 / 出口) | acquire | D21 |
| epoch fetch_add | release | D22 |
| epoch 快照 load / wait | acquire | D21 |
| notify_all | (非内存操作) | D22 |

- **result 的 acquire 接 Engine 索引序**:leader 程序序"fdatasync → result 回填
  (release)";调用者 acquire 读到结果返回后在同一线程写内存索引——该 release/acquire
  链保证**帧持久化先行于索引写入**,级别 1/3"WAL 落盘 + 内存 → 返回"的顺序承诺在并发
  版本里就靠这一对内存序成立(提交组连接 Engine 语义的最后一环)。
