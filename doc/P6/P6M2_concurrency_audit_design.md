## 1. 目标与范围

### 1.1 定位：收口验证，不是实装

M2 与 M1 性质相反。M1 已经把提交组协议、批处理管线、全部失败语义（单一失败出口 +
结果矩阵 + 状态语义 + 不进全拒态）写进代码并测过（157/157，含 TSAN 自查）。M2 不碰
机制，只做验证与对账，确认提交组改造没有破坏既有机制。

### 1.2 净增量与防回声判据

**净增量四样**：① 对账文档（主体，含失败不变量对账、运营口对账、恢复对账）；
② 故障处理三档的代码标记（运行时故障防御代码加注释）；③ 检测器矩阵的正式签收；
④ 向前同步（项目假定、待梳理点关闭、文档语言清理）。**没有新机制、没有新功能测试。**

**防回声判据（P6M2-D2）**：M2 产出"判断与标记"，不产出"机制与测试"。任何想在 M2
写的新机制代码或新功能测试，一律退回——若是 M1 漏的实装，算 M1 补丁；若只是验证 M1
已有的，用审查或重跑。两条推论：① 失败后不变量对账是纯审查（按"假定软硬件不故障"
政策，失败路径无注入手段，对账 = 读代码对照不变量、写结论，不产出测试）；② 三不变量
（I1/I2/I3）证明不重做（M1 §4.5 已给，需要时只引用）。一条反向保护：对账若发现 M1
真缺陷，就地修 M1 + 在 M2 注明，不甩锅、不另加机制（本次预审未触发）。

### 1.3 范围逐项归类

| M2 范围项 | 归类 | 增量 |
|---|---|---|
| 失败后状态不变量对账 | 新审查 | 对账表（§4） |
| 故障处理三档落地与标记 | 新代码 | 注释标记（§3，轻） |
| 运营口互动对账 | 新审查 | 对账表 + P7 记账（§5） |
| 恢复对账 | 重跑确认 | 论证 + 既有用例（§6，零新测试） |
| 检测器矩阵收口 | 重跑确认 | 矩阵签收（§8，零新代码） |
| 全拒态终裁 | 新审查 | 裁决（§7） |

实质工作集中在前三项；恢复与矩阵是"跑一遍确认"；只有故障标记有真正的代码改动（且很轻）。

### 1.4 独立里程碑

M2 保留独立、不并入 M3（P6M2-D3）。理由：M2 是正确性收口、M3 是收敛（性能锚点基线 +
状态同步），两类工作不同性质；M2 是 M3 的前置闸门（没验对就采基线、宣告完成没意义）；
薄里程碑在 cabe 有先例（P5M7、P4.5M3）。"对账确认无事"本身就是有价值的交付结论。

## 3. 故障处理边界（D4/D5/D6）

### 3.1 三档政策

cabe 假定其依赖的软硬件环境正常、不发生运行时故障（项目级假定，见 ROADMAP）。故障处理分三档：

1. **明显故障必须健壮**：设备打开失败、内存分配失败、参数非法三类——代码必须判断并
   干净处理（返回错误码 / 拒开）。这类简单故障代码检查容易做到，必须保证健壮性。
2. **运行时软硬件异常一概不投入**：`WriteAt`/`Sync`/`ReadAt` 等运行时 I/O 故障，按
   假定不发生；活写路径已有的防御性返回是兜底，**未经故障注入测试**，完整性仅靠代码
   审查，代码中以"运行时故障未测"标出。
3. **资源状态不是故障**：`kWalFull`（环写满）是正常资源状态，照常真测（M1 用例 E
   填满环触发）。

### 3.2 明显故障检查盘点

三类明显故障的检查在代码中已齐全，M2 零新增：

| 位置 | 明显故障 | 处理 |
|---|---|---|
| `Wal::Open` | `opts` 空指针 / 设备打开失败 / 容量不足 / 缓冲分配失败 | `kEngineInvalidOpts` / 原样返回 / `kDeviceTooSmall` / `kIoBase` |
| `Wal::Recover` | 设备未开 / 重复调用 / 缓冲分配失败 | `kEngineInvalidOpts` / `kIoBase` |
| `Wal::WriteWal` 入口 | key 超长 / 缓冲未就绪 | `kWalKeyTooLong` / `kWalWriteFailed` |

### 3.3 运行时故障标记

**标记范围**：只标三处"防御性写好了但未测完整"的活写路径运行时故障返回点：
提交组批末 `dev_.Sync`、`WriteWindowSpan` 的 `WriteAt`、`Flush` 的 `dev_.Sync`。
恢复路径的读写失败（处理是拒开 / 中止，保守且完整正确，又是 P5M6 范畴）**不标**——
给已经做对的路径贴"不完善"反而误导。

**标记形态**：一处集中说明 + 各点一行可检索短标记，不逐点重复政策。集中说明放提交组
区段开头（讲一次政策、指向项目假定）；三个失败返回点各加一行"运行时故障未测（见区段
顶注）"，方便以后一条命令查全。

## 4. 失败后状态不变量对账（D7）

### 4.1 失败场景（对账的行）

`ProcessBatch` 的失败出口归并为五个场景；S4（`Append` 失败）提交组下入口已挡 `cur_buf_`
空、实际不可达，仅作代码防御分支，不占对账行。

| 场景 | 触发 | 失败后状态 |
|---|---|---|
| S1 批首惰性重开仍满 | `kWalFull` | 本批未写未编码；状态原样 |
| S2 中段写出失败 | `WriteAt` 错 | 状态未动——前序帧滞留缓冲（seq 已耗）、`cur_off_` 不前进 |
| S3 中段写出后开窗仍满 | `kWalFull` | 中段已推进、`window_bytes_=0`；本帧未编码 |
| S5 末段写出失败 | `WriteAt` 错 | 状态未动——本批帧滞留缓冲、`cur_off_` 不前进 |
| S6 批末 fdatasync 失败 | `Sync` 错 | 已推进、不回滚——帧在盘未确认，`cur_off_` 已前进 |

写失败类（S2/S5）：`WriteOutWindow` 在 `AdvanceAfterWrite` 之前返回错误，故"状态一动
不动"。fsync 失败类（S6）：写已成功、`AdvanceAfterWrite` 已执行，故"已推进、不回滚"。

### 4.2 不变量（对账的列）

四列，对齐到 M5/M6 既有不变量编号：

- **C1 活区间紧凑 + seq 稠密**（M5-③∧④ / M6 走读单判据）：失败不在盘上留块对齐空洞、
  不在 seq 序列挖洞（已分配的 seq 必有对应帧存在于缓冲或盘上）。最关键的一列。
- **C2 推进单调 + 恒留一块**（M5-① / D5）：`cur_off_` 失败后不回退；恒留一块边界不破。
- **C3 撕裂安全**（M2 撕裂安全模式）：滞留 / 已写段下一批同偏移同字节重写，崩溃安全。
- **C4 运行态自洽**（实现态）：`window_bytes_`/`n_frames_`/`cur_off_` 三者失败后一致，
  下一批准入逻辑正确接续。唯一非盘面、但失败接续的正确性全靠它。

### 4.3 对账表（5×4 全过）

| 场景＼不变量 | C1 紧凑+seq稠密 | C2 推进单调+恒留一块 | C3 撕裂安全 | C4 运行态自洽 |
|---|---|---|---|---|
| **S1** | ✓ 未编码、`seq_next_` 未动、盘面未变 | ✓ `cur_off_` 未动；`window=0` 即恒留一块满态 | ✓ 无写动作 | ✓ 满态标准姿态、惰性重开 |
| **S2** | ✓ 前序帧 seq 已耗但在缓冲；盘面未变无新洞 | ✓ `cur_off_` 未进 | ✓ 滞留帧下一批同字节重写 | ✓ 状态未动；下一批先重试写出 |
| **S3** | ✓ 中段必整块→盘上紧凑；本帧未编码无新 seq | ✓ `cur_off_` 已前进；`window=0` 满态 | ✓ 留窗块下次同字节重写 | ✓ 推进后三者一致 |
| **S5** | ✓ 同 S2 | ✓ `cur_off_` 未进 | ✓ 滞留帧下一批重写 | ✓ 状态未动；下一批继续填充、seq 连续 |
| **S6** | ✓ 帧已写到盘、紧凑、seq 连续 | ✓ `cur_off_` 已前进 | ✓ 帧在盘未确认→下一批 fsync 顺带确认 | ✓ 推进后三者一致 |

**结论：20 格全部成立，无反例。** 两处关键依据：

- **S2/S5 的"状态未动"是真的**：`WriteOutWindow` 在 `WriteWindowSpan` 失败时先返回、
  不调 `AdvanceAfterWrite`，故 `cur_off_`/`n_frames_`/`window_bytes_` 一个不变，滞留帧
  连同已耗 seq 原样留缓冲——这是 C3/C4 成立的代码依据。
- **S6 的"推进不回滚 + 幽灵写"不破不变量**：帧已 `WriteAt` 成功（在盘），写者虽被告知
  失败但帧可能恢复出来，属文档化的幽灵写语义（"多恢复未承诺数据"），不违反 C1~C4
  （它们管"不破坏盘面一致性"，不管"是否多恢复"）；后续撕裂交 M6 容差，自洽。

发现反例的处理（D2 反向保护）：就地修 M1 代码 → 重跑回归 → 在本对账表注明，不另加机制。
本次预审未触发。

## 5. 运营口互动对账 + P7 记账（D8/D9）

### 5.1 静止前提

提交组的窗口状态（`cur_off_`/`n_frames_`/`window_bytes_`/缓冲/`head_off_`）按钥匙制 I1
只有在位 leader 能碰；运营口也碰这些状态。对账的根本问题是：运营口运行时会不会有
leader 正在飞？

**P6 下不会，运营口永远看到静止的提交组**，两根支柱：

1. **公开接口单线程**：五个口子全从 Engine 调，且都和 `Put`（进而 `WriteWal`）同线程；
   `DoSnapshot` 在 `WriteWal` 返回之后才触发，而 `WriteWal` 返回时 leader 已卸任、本
   写者已读到结果——提交组完全静止（栈空、无人在位、无等待者）。
2. **并发测试驱动契约（P6-D9）**：直打 `Wal` 的并发测试只做"多写者并发 `WriteWal`"，
   不并发调任何运营口。

这条静止前提不靠任何锁，靠这两条外部约束。它正是 P7 会破的原因（两条约束届时都没了）。

### 5.2 五口子逐项对账

| 口子 | 静止态下安全性 | 细微点 |
|---|---|---|
| **Flush** | ✓ | 同步档（1/3，提交组活）下是空操作（每批已 fdatasync、无待刷），`covered_seq` 捕获正确；攒批档下提交组不参与 |
| **reclaim_boundary** | ✓ | 纯读 `cur_off_`，静止态下即批末持久边界，与 `covered_seq` 物理孪生（提交组盘面与 P5 攒批同构） |
| **ReclaimUpTo** | ✓ | 唯一动 `head_off_` 处，几何校验防回退、只前进——**补齐议题 3 deferred 的 head_off_ 单调性** |
| **SetWalLevel** | ✓ | 切档交接靠两档共用同一套缓冲 / 窗口表示法（M1 §5.5）；级别读取无撕裂（单线程串行） |
| **Close** | ✓ | 释放缓冲的安全依赖无 leader（静止前提）——若有 leader 在飞，拔掉缓冲会崩 |

**结论：五口子在静止前提下全部安全**，无需加锁或改代码，安全性与 P5 单线程时同源。两处
收获：ReclaimUpTo 补齐了议题 3 deferred 的 `head_off_` 单调性；SetWalLevel 切档干净的
根因是 M1 §5.5 的"一套表示法"。

### 5.3 P7 并发记账

公开接口多线程后，静止前提两根支柱皆塌，三处竞争冒出：

1. **运营口 × leader 抢窗口状态**：快照的 Flush / reclaim / ReclaimUpTo 撞上正在写环的
   leader，同碰 `cur_off_`/`head_off_`/`n_frames_`/缓冲；
2. **`SetWalLevel` × `WriteWal` 撕裂派发**：改级别撞上入口读 `sync_level()`，攒批写者
   与提交组 leader 混跑同碰窗口状态；
3. **`Close` × `WriteWal` 释放后使用**：释放 `cur_buf_` 撞上 leader 正在写、崩溃。

**还法（记明，免得 P7 走偏去加锁）**：靠 P7 的 reactor 重建串行——每设备一个 reactor，
该设备的全部 `Wal` 操作（写入 + 五运营口）投递进同一事件循环、被它一件一件处理，P6 的
静止前提由 reactor 重新建立。**P7 不需要给 `Wal` 加任何锁**，reactor 单线程事件循环本身
就是串行点，与 ROADMAP P7"无任何 mutex"一致。本条细化 M1 §14"运营口跨操作并发归 P7"
那一句，不另开新债（并发救援协调、"重试必成"失效 M1 §14 已记，此处引用不重复）。

## 6. 恢复对账（D10）

### 6.1 证据与判据

恢复对账的证据 = 两组既有用例，零新测试：

- **`recovery_test` 8 例（Engine 级）**：`Put → Close → recover → 验值`，确认经提交组
  写入的数据能被恢复链正确恢复。M1 之后 `Engine::Put` 内部就走提交组（批大小 1），故
  这 8 例已端到端跑过"提交组写入 → M6 恢复"，它们在 M1 全绿就等于这条链已验证。
- **并发用例 F（Wal 级）**：并发写 → recover → 续写 → 再 recover，确认并发环能正确
  重放、续写衔接成立。

判据：这两组在 §8 收口矩阵里全绿即过，不单独跑。后端无关性：WAL 恢复链走 `RawDevice`、
不经 IoBackend，双后端下恢复扫描执行同一份代码，差异只在验值那一步（value 走 IoBackend）。

### 6.2 单线程等价论证

为什么不用新测试也能断定 M6 不受提交组影响，四步：

1. Engine 单线程 ⇒ 每次写都是批大小 1（自任 leader、无竞争）；
2. 批大小 1 的盘面与 P5 逐帧路径逐字节等价（引用 M1 §5.3/§5.5，不重证）；
3. Engine 产出的整盘面（WAL + 快照 + 数据盘）= P5 的盘面；
4. M6 恢复链作用于盘面，盘面相同则恢复行为相同 ⇒ 提交组改造经 Engine 驱动时 M6 不受影响。

派生结论：唯一并发处是 Wal 层并发测试，用例 F 已覆盖并发环恢复。两个面（Engine 级单线程
盘面等价 + Wal 级并发环）都覆盖，恢复对账完备。

## 7. 全拒态终裁（D11）

**全拒态**指：某次写失败后提交组进入"中毒"状态，此后所有写一律无条件返回
`kWalWriteFailed`、不再尝试。代码事实：M1 现状就是"不进全拒态"（失败只设局部变量、
无持久中毒标志）。

**终裁：维持不进全拒态。** 两条判据都指向它：① 议题 3 对账无反例——满足 M1 原判据
"找不到失败后破坏不变量的路径"，继续服务无正确性风险；② 项目政策"假定软硬件不故障"
——全拒态是为假定不发生的故障加持久状态机、过度设计。

落地：**零代码改动**（现状已是"不进"，要"进"反而才需新加中毒标志）。故障吵闹行为如实
记录——假定不故障下不会发生；万一设备永久故障，行为是"每次写单独失败、滞留重试，吵闹
但不毒化、不丢正确性"（M1 §8.2"吵闹换自愈"的延续），可接受；留一句：未来若 cabe 超出
学习 demo、需工业级故障处理，全拒态可重新评估。关闭 P6 README 待梳理点①。
