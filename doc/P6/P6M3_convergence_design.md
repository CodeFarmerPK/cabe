## 1. 目标与范围

### 1.1 定位：收敛 + 微基准代码，产品代码冻结

M3 是 P6 的收敛里程碑。产品的核心功能在 M1（提交组机制）、收口在 M2（对账 + 故障标记 +
检测器矩阵）已完成；M3 只做收敛与性能测量准备，**不动 engine/wal/snapshot 任何产品代码**。
三块产出：

1. **全量微基准代码**（测量代码，只在 `bench/`）；
2. **薄收敛稿**（本文档）；
3. **状态同步**（根 README / P6 README / 各稿状态字段）。

### 1.2 P6 = M3 完全收尾 / M4 名义挂靠

**M3 完成即 P6 完成，与 P4/P5 同级地完全收尾**——P6 的全部代码、测试、收尾（含状态同步、
P6 完成标记）都在 M3 做完，**P6 不依赖 M4**。

**M3 测试政策（仅作正确性证明、数据不入 M3）**：M3 可跑 test / 覆盖率 / bench 三类测试，但
只为证明正确性（改动不破坏正确性、bench 能产出数字）；**测试数据与产物文件不在 M3 体现**
（不归档基线 JSON、不入库任何测试数据）。具体：

- **test / 覆盖率**：M3 不改产品代码，正确性沿用 P6M2 §8.1 签收（157/157 sync、167/167
  io_uring、TSAN 20 轮、覆盖率），无需重采；
- **bench**：代码编译通过 + 冒烟跑通（证明能产出数字即可），数字丢弃、不固化。

**M4 只是名义上挂在 P6**——因体量太小挂在 P6 尾部，实质是个**独立的测试里程碑**：在标准硬件
上正式采集锚点数据并上传 / 归档，供 P7 及后续阶段对比。它不构成 P6 的组成门槛，P6 收尾与
M4 无关。本收敛稿不含基线数字（数字由独立的 M4 产出）——这不影响 P6 收尾：P6 的交付是机制 +
可用的 bench 工具，数字是给后续阶段用的独立产物。

### 1.3 范围护栏（防越界判据）

三条护栏，越出任一即视为越界：

1. **不碰产品代码**：只动 `bench/` + `scripts/mkloop.sh` + 文档；engine/wal/snapshot 冻结。
   想加计数器测合并率＝破坏冻结，退回。
2. **微基准只围绕 P6 现有功能、不预判未来**：覆盖 P6 截至当前能做的（Put/Get/Delete ×
   WAL 级别、多线程 Wal 吞吐）；不预埋 P8 对齐缓冲 / P9 点查 / P10 后端 / P11 多设备等
   未来测试点（各阶段自带）。
3. **度量口径 = 吞吐（= IOPS，固定 1 MiB）+ 单线程自带延迟**：合并率、快照时长随规模等
   推迟到发布后；不加任何需要产品代码支持的指标。

## 3. 单线程微基准（D4/D5/D6）

### 3.1 覆盖矩阵：12 条（Put/Get/Delete × WAL 级别 1/2/3/4）

形式统一的 4×3 矩阵，对称、整齐、不用每次解释裁剪逻辑。冗余的 5 条（Get×4 应趋同、
Delete 1≡3/2≡4）不是浪费——它们是**交叉验证证据**：跑出来若 `Get/1≈…≈Get/4`、
`Delete/1≈Delete/3`、`Delete/2≈Delete/4`，就经验性印证"Get 读路径与 WAL 级别无关、Delete
无 value 故只随同步/攒批变化"；万一某处意外不等，是值得查的信号。报告里写一句**预期等价**,
把 12 条解读为"印证预期"而非 12 个独立数。

### 3.2 参数化机制与接线

用 google-benchmark `Args`：三个微基准各 `->Arg(1)->Arg(2)->Arg(3)->Arg(4)`，
`state.range(0)` 读级别，级别自动进输出名（`BM_Put/1`…`BM_Put/4`），单次运行产出全 4 档。
接线四处：

1. `MakeOpts(create, level)`：加 `level` 参数，`opts.wal_level = static_cast<WalLevel>(level)`；
2. `SetUp(state)`：读 `state.range(0)` 作级别开引擎；
3. 重开 / 重填路径：用同一级别（别掉回默认 3）；
4. 级别合法性守卫（`IsValidWalLevel`），防手滑传错。

### 3.3 度量口径

- **Put / Get**：`SetBytesProcessed`（现成）→ bytes/sec = 吞吐（= IOPS × 1MiB）；
- **Delete**：`SetItemsProcessed`（现成）→ ops/sec = 删除 IOPS（无 value，按操作数算）；
- **延迟**：google-benchmark 默认报每次迭代耗时（`Time`），即单次操作延迟（固定 1MiB 下与
  吞吐互为倒数）——白送的副产品，**零新指标代码**。
- **计时基准 = 墙钟（`UseRealTime`，三个 bench 都加）**：Put/Get/Delete 都是 I/O 密集——每次
  操作 1MiB 写 + fdatasync，时间几乎全在 D 态等盘、CPU 时间近零。若按 google-benchmark 默认的
  **CPU 时间**满足 `min_time`，框架会以为"操作超快"而疯狂加大迭代数（永远凑不够 CPU 时间）→
  跑不完、还把设备写满触发重开；且 `bytes/items_per_second` 按 CPU 时间算会严重虚高。故三个
  bench 均 `->UseRealTime()`：迭代数按墙钟有界、吞吐 = 真实 bytes/墙钟（与 §4 多线程 Wal bench
  同理）。**没有这一条，I/O 密集 bench 会空转数小时**（实证：默认 CPU-min_time 跑满 32G 反复重开）。
  附带后果：墙钟有界后单配置默认仅跑数百次操作（< §5 快照阈值对应的 ~8192 次），周期快照不触发；
  **M4 采集时须取较大的 `--benchmark_min_time`（墙钟秒）让单配置操作数 > ~8192**，把周期快照开销
  计入（兑现 §5 的 1M 阈值意图）——具体值与"够大设备不写满"的平衡留 M4 定。
- **M4 注意：异步档（级别 3/4，value 不 FUA）的 writeback 伪影**（M3 io_uring 冒烟实测发现）：异步
  写不立即落盘，在 loop/稀疏设备上积压后由内核后台 writeback 集中刷出、引发偶发几十毫秒停顿，被
  均值放大——表现为 Put/3·4、Delete/4 偏慢、Delete「2≡4」量测不显。**这是宿主机伪影、非引擎缺陷**
  （引擎正确性由 167/167 测试独立保证；Delete 各档语义相同，差异来自重填 Put 的异步 value 写积压）。
  M4 正式采集须规避：配置间 `fsync`/掉缓存、稳态预热，或直接在真实 NVMe 上跑。

性能测试只测**连续稳态**读写删；引擎关闭重开（写满后 `Close + Open(create)` 重新格式化）
**不属于性能测试场景**。现有 `BM_Put` 在小设备上每 ~12 次 Put 触发一次重开尖峰，会污染
"最大性能"。M3 的解法不是 `PauseTiming` 排除，而是**用足够大的设备让性能测试永不写满 →
永不触发重开**（见 §5）。原"PauseTiming 排除重开"方案作废。周期快照是**真实负载**、保持
计时（与重开这种测试假象切开）。

## 4. 多线程 Wal 吞吐微基准（D7/D8/D9）

### 4.1 形态与度量

新文件 `bench/wal/wal_concurrency_bench.cpp` + target `bench_wal_concurrency`（链 `cabe::wal`）。
**直打 Wal、不经 Engine**——提交组的并发只在 Wal 模块层存在，Engine 接口单线程从它测不出
并发收益。测**聚合 frames/sec**（`SetItemsProcessed`、框架跨线程求和），报告注明
**× 1MiB = 等效 value 提交吞吐**。主看 **1/2/4/8 写者的吞吐曲线**——这是 group commit 的
成绩单（1 写者退化、自任 leader 无合并 → 多写者批越大、fsync 越摊薄、聚合吞吐递增）。

只跑**同步档（级别 3）**：提交组只在同步档生效；攒批档无提交组、直打多线程不安全（归 P7）。
故不按级别参数化，固定同步档、只按写者数参数化。

### 4.2 线程模型

google-benchmark `->Threads(1)->Threads(2)->Threads(4)->Threads(8)`，框架起 N 线程跑循环体、
聚合 ops/墙钟。**共享 Wal 开一次**：`state.thread_index()==0` 在 `SetUp` 开 Wal；安全性靠
框架的计时屏障——所有线程的 `SetUp` 必先各自跑完才到屏障，屏障凑齐 N 个才放行，且屏障提供
happens-before，故 0 号线程开的 Wal 在其余线程过屏障后可见。（不用 `std::call_once`——它
不可重置、跨 1/2/4/8 多配置会失效；`thread_index()==0` 随每配置 `SetUp` 重跑天然重置。）
否决手动 std::thread 方案（输出脱离框架、不统一）。循环体极简：构造最小 `WalEntry`（常量
key、占位 block，吞吐不在乎内容）调 `WriteWal`。

### 4.3 容量与边角

直打 Wal 无回收（无 Engine/快照来 `ReclaimUpTo`），WAL 环只进不出、会写满。三管齐下防填满：

1. **每配置 fresh 环**：`SetUp`（0 号线程）用 create 开全新空环；
2. **bench WAL 设备 1 GiB**（≈ 8M 帧余量；见 §5）；
3. **运行时长有界**：M4 跑时用默认/短 min-time，保证单配置总帧数远 < 环容量。

只报吞吐、**无合并率**（合并率需 Wal 暴露计数器 = 改产品代码，不做）。

## 5. bench 设备方案（D6/D9）

bench 用**独立镜像文件**（`cabe_bench_{data,wal,snapshot}.img`），与测试小设备组
（`cabe_test_*.img`）分开、互不干扰——测试套件继续用小设备（撞墙类用例按 `RingBytes()`
填满整个小环，放大会拖垮它），bench 用大设备（永不写满 → 永不重开 → 稳态干净）。

`scripts/mkloop.sh` 加 `create-bench` 预设（沿用现有 cleanup→create 流程，先解绑再建）：

| bench 设备 | 大小 | 推算 |
|---|---|---|
| value | **32 GiB** | 每次写 ≤ 20G < 32G，留 12G 余量、永不写满 |
| snapshot | **128 MiB** | 32G 满载 = 32768 key；快照 = 32768 × 128B × 双槽 ≈ 16 MiB，128M 为 8× 余量 |
| WAL | **1 GiB** | 多线程微基准无回收，环 ≈ 8M 帧余量；容量校验 `环 ≥ 阈值×2` 对 1M 阈值轻松过 |

镜像为稀疏文件，创建瞬时、不实占盘，只在写入时占用——开大零成本。每次 bench 写入 ≤ 20G
作安全上限（实际每配置 `SetUp` create 重置、单配置写入由迭代数定、远小于 20G）。bench 快照
阈值 **1M**（让快照在 ≤20G 写入内真实触发几次、计入吞吐；快照是真实负载，与重开假象不同）。

引擎微基准的写满重开代码在大设备下永不触发——保留作死安全网即可（加注释说明不触发），不必
特意删除。

### 6.1 里程碑摘要链回

| 里程碑 | 一句话摘要 | 设计稿 |

| M1 | WAL 同步档"每帧一写一刷"改造为 leader/follower 提交组：并发写者的帧由轮值 leader 合并成批、批末一次 fdatasync；盘上格式与恢复链零改动（D1~D28） | P6M1_commit_group_design.md |
| M2 | 并发收口：失败后不变量对账（5×4 全过）、运营口互动对账、恢复对账、故障三档标记、检测器矩阵签收（D1~D13） | P6M2_concurrency_audit_design.md |
| M3 | 收敛 + 全量微基准代码 + 状态同步（本稿，D1~D13） | 本文档 |

### 6.2 提交组终态

协议要点：栈上 `WriterNode` + 无锁 MPSC 提交栈；"由空变非空"者自任 leader；卸任三步
"让位→复查→条件再任职"（四操作 seq_cst）；窗口状态钥匙制（仅在位者可碰）。**盘上格式
（128B 帧、4K 整块、环形推进、变体 Y）与 M6 恢复链零改动**——leader 批量落盘的盘面与 P5
逐帧路径逐字节等价（详见 P6M1 §5）。对外仅两条行为变化：`kWalFull` 经批处理回填（语义不变）、
同步档并发 `WriteWal` 从未定义变为受协议保护。

### 6.3 不变量汇总

- **M1 三不变量**：I1 钥匙制（在位标志至多一人持有，窗口/缓冲/盘面随标志走）/ I2 恰好一次
  （每节点被取批一次、回填一次）/ I3 无遗弃（栈非空则有限步内必有取批）。证明见 P6M1 §4.5。
- **M5/M6 既有不变量被提交组完整保持**：seq 稠密、活区间紧凑、撕裂安全、走读恢复判据——M2
  §4 的 5×4 对账（20 格全过）已确认。
- **一句话定调：提交组未破坏任何既有不变量**（引 M1 §4.5 + M2 §4，不重证）。

### 6.4 收口验证实证

引 P6M2 §8.1：7 配置检测器矩阵全绿（同步四档 157/157、io_uring 三档 167/167）+ 多线程并发
测试 TSAN 下 20 轮 0 失败 + 覆盖率（行 86.1% / 函数 98.2% / 分支 50.8%）。

### 6.5 后端策略：冻结 sync、锚点转 io_uring（D14~D17）

sync 后端是 P0~P6 早期为快速落地走的同步路线；从 P7 起 cabe 在 io_uring 上引入异步、无锁、
多线程，这些技术的收益主要在 io_uring 上兑现。据此截至 P6 定下后端策略：

**冻结分粒度（D14）**——分三层，前两层冻、第三层留：

- 冻结 sync 的**开发/优化**：不再加特性、不做 sync 性能工作；
- 冻结 sync 的**性能基准**：锚点只在 io_uring，sync 不再出基准数字；
- **保留 sync 的正确性回归测试**：① 作 io_uring 的差分参照（同批操作两后端应产出逐字节
  相同的盘上状态）；② TSAN 与 io_uring 不兼容（P4M3 既定），数据竞争证据只能由 sync+TSAN
  产出——保留成本近零、保险价值高，P7+ 大改 io_uring 时尤甚。

**性能锚点 = io_uring（D15）**——P6 给出的基准即 io_uring 的基准，确切含义是
**io_uring 后端、同步（提交即等待）用法、单线程引擎**（io_uring 当前为 submit-and-wait，
见 `io/uring/io_uring_backend.cpp`）。这正是 P7 异步化的干净起跑线：同后端、未异步，P7 的
提升能纯粹归到异步技术。提交组**后端无关**（写在 `wal.cpp`、走 IoBackend 抽象、批末
`dev_.Sync()` 一次 fdatasync），故 io_uring 锚点测的就是 P6 真正做的提交组，不存在"招牌
特性在 io_uring 上不生效"。bench 测量代码零改动（后端是编译期选项，M4 采集时 configure
`-DCABE_IO_BACKEND=io_uring`）。

**取消默认后端、`--backend` 必填（D16）**——顶层 CMake 去掉 `CABE_IO_BACKEND` 默认值，后端
一律显式指定、缺省即报错；影响 CMake + 三个脚本（`run-tests`/`run-coverage`/`run-bench`）。
**已实装**：CMake 缺省 `-DCABE_IO_BACKEND` 即 `FATAL_ERROR`，三脚本 `--backend` 必填（见 §11）。

**io_uring 并发签收口径（D17）**——io_uring 不能跑 TSAN，故不另立"io_uring TSAN"签收；其
并发正确性由两条支撑：io_uring 功能并发测试全绿（167/167，P6M2 §8.1）+ 提交组后端无关
（仅 leader 单线程访问 `dev_`、follower 仅等待，io_uring 后端无需自身线程安全）使 sync+TSAN
的数据竞争证据可迁移到 io_uring。这反过来坐实 D14"保留 sync 回归测试"的必要性。
