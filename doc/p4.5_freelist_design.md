# Cabe P4.5 — FreeList 改造设计稿

> 阶段定位:P4 io_uring 后端收尾(2026-05-14)→ P5 WAL 之间的独立小阶段
> 目标:FreeList 三容器轮换 + 严格升序分配 + 异步 sort,为 P5 WAL recovery 接口预留 hook
> 范围:仅 `storage/free_list` 模块改造,不触动 Engine 业务逻辑、不引入持久化
> 状态:v1.0 定稿(2026-05-15),待实施

---

## § 1 阶段定位

### 1.1 与现有阶段的关系

```
P3(IoBackend 抽象,✅)
  ↓
P4(io_uring 后端 + registered buffer,✅ 2026-05-14)
  ↓
P4.5(本阶段:FreeList 三容器改造)
  ↓
P5(WAL + 崩溃恢复)
  ↓
P6(reactor 多线程)
  ↓
P7-P9(B+ 树 / scatter-gather / SPDK)
```

P4.5 独立成阶段(不并入 P5)的理由:

1. **数据结构改造的风险隔离**:freeList 是核心分配逻辑,与 WAL 同时改造会让 bug 难以定位
2. **接口稳定先于持久化**:P5 WAL recovery 依赖 freeList 的 `RebuildFromActive` 接口,先固化后接入
3. **与"先落地后优化"原则一致**:逐档独立交付,逐档独立 bench 归档
4. **bench 价值独立**:freeList 改造对 NVMe 写入模式的影响可单独验证

### 1.2 P4.5 范围

**做**:
- freeList 数据结构从单 list 改为三容器轮换(freeList + 双 recycle)
- 严格升序分配(每次分配的 BlockId 是 freeList 当前最小者)
- 异步 sort 工作线程,业务路径几乎零阻塞
- TRIM 集成(Delete 路径同步 `ioctl(BLKDISCARD)`)
- Stats 观察接口
- `RebuildFromActive` 接口(P5 WAL recovery 用,P4.5 实现 + 单测)

**不做**:
- 不引入 freeList 自己的业务路径锁(继续依赖 Engine 大锁,D-8)
- 不引入持久化(P5 WAL)
- 不引入 freeList 切换的 WAL 帧(freeList 状态可从 chunkIndex 推导)
- 不优化小 chunk(< 256 KiB)场景,文档说明推荐 chunk 大小
- 不引入后台异步 TRIM(同步发,失败 WARN)

## § 2 设计动机与 cabe 场景假设

### 2.1 cabe 工作负载特征

cabe 是大 value KV 存储引擎,目标场景:

| 场景 | 写入模式 | 删除模式 | 修改频率 |
|---|---|---|---|
| 监控视频存储(NVR / VMS) | 顺序写入流 | 批量按时间窗口清理 | 几乎零 |
| 大对象存储(类 S3) | 用户上传 | 零散偶发 | 几乎零 |
| 备份归档(Veeam / Bacula 类) | 周期性顺序 | 批量按 retention | 零 |
| HPC 中间结果 | 任务顺序写 | 批量任务结束清理 | 零 |

**共性**:写入接近 100% 顺序、删除主要批量、修改极少。

不适合 cabe 的场景:
- 通用 OLTP KV(RocksDB 风格,小对象 + 高随机性)
- 高频更新场景(每秒大量 Put 覆盖)

### 2.2 严格升序分配的设计目标

NVMe 实测数据(企业级 PCIe 4.0,1 MiB 块):
- 顺序写带宽:6.5 GB/s
- 随机写带宽:6.0 GB/s
- 差距:~8%

虽然 1 MiB 块下顺序/随机差距已小,但 cabe 主场景下:
- 批量删除产生**连续 LBA 段**,升序与 LIFO 在连续段上等价(NVMe 对 LBA 方向不敏感)
- 段间过渡:升序使段间跳跃方向可预测,SSD FTL 内部优化略友好
- 整体性能差距:< 2%

**严格升序的真正价值是工程哲学上的"行为可预测"**,不是性能上的硬性需求。在 cabe demo / 学习项目定位下,这种可预测性具有教育价值,且工程成本(异步 sort + 三容器)可控。

### 2.3 异步 sort 的必要性

切换时 sort 的工作量:
- 40T 设备稳态下,sort 范围约 10M BlockId
- 同步 sort 时间:~500 ms(单线程持锁,严重阻塞)
- 异步 sort:业务零感知,只有 CompleteSwitch 的 O(1) idx 互换持锁(< 1 μs)

**异步 sort 是"接近零阻塞"的关键**。代价:引入后台线程基础设施,约 700 行新代码。

## § 5 架构总览

### 5.1 三容器数据结构

```cpp
class FreeList {
private:
    std::vector<BlockId> containers_[3];   // 三个固定容器,生命周期 = FreeList 生命周期

    uint8_t free_idx_;        // 当前作为 freeList 的索引(分配源)
    uint8_t active_idx_;      // 当前接收 Release 的索引
    uint8_t inactive_idx_;    // 当前 inactive 索引(等待 sort 或保持残余)

    // 不变式:三个 idx 互不相同,值在 {0, 1, 2} 中
};
```

**容器角色**:
- **freeList**(`containers_[free_idx_]`):分配源,**降序存储**(pop_back 取最小,实现严格升序分配)
- **active recycle**(`containers_[active_idx_]`):接收 Release(无序累积)
- **inactive recycle**(`containers_[inactive_idx_]`):
    - 切换期间:已被 sort 走的容器(临时空)
    - 切换完成后:接收旧 freeList 残余,等下一轮被 swap 成 active

### 5.2 角色互换机制(idx swap,O(1))

切换时**通过 idx 互换实现角色变换,数据不动**,避免 memcpy 阻塞:

```
StartSwitch(业务线程,Engine 锁内):
    swap(active_idx_, inactive_idx_)
    旧 active 容器(刚 move 到 sortTask)→ 现在变 inactive(空)
    旧 inactive 容器(可能含上次残余)→ 现在变 active,接收新 Release

CompleteSwitch(业务线程下次入口检测 sortDone 后):
    swap(free_idx_, inactive_idx_)
    旧 inactive 容器(已 sort 完)→ 现在变 freeList
    旧 freeList 容器(含残余)→ 现在变 inactive
```

**残余生命周期**:
- 第 1 轮切换:残余进 inactive
- 第 2 轮切换:残余跟随 swap 进 active,继续接收 Release
- 第 3 轮切换:active(残余 + 新 Release)整体送 sort,成为新 freeList

残余等待 2 轮才能被分配,**这是 O(1) 切换的代价**,40T 稳态下可接受(见 § 13.2 性能特征)。

### 5.3 状态机

```
Running ── freeList 阈值 / 对称水位触发 ──→ Switching
   ▲                                              │
   │                                              │
   └────── 业务线程下次入口 + sortDone ◀──────────┘
            (CompleteSwitch)
```

`state_` 用 `std::atomic<State>` 管理,只有 2 种值。

### 5.4 并发模型

```
业务线程(持 Engine.unique_lock):
    Allocate / Release / StartSwitch / CompleteSwitch
        ↕
    worker_mu_(次级锁)
        ↕
后台 sort worker 线程(不持 Engine 锁):
    cv.wait → sort(锁外)→ 回填 sort_result_ → notify
```

**锁层级**:Engine 锁 > worker_mu_,单向获取,无死锁可能。

**TSAN 友好性**:
- 跨线程数据(`sort_task_` / `sort_result_`)在 `worker_mu_` 下访问
- 状态标志(`state_` / `sort_done_` / `stop_`)用 `std::atomic` + acquire/release
- 容器数据完全在 Engine 锁内访问,无跨线程访问

## § 6 接口契约

### 6.1 公开接口(`storage/free_list.h`)

```cpp
class FreeList {
public:
    FreeList();
    ~FreeList();  // 析构:停止后台线程,join

    // 禁止拷贝/移动(持有线程和互斥量)
    FreeList(const FreeList&) = delete;
    FreeList& operator=(const FreeList&) = delete;
    FreeList(FreeList&&) = delete;
    FreeList& operator=(FreeList&&) = delete;

    // === 生命周期 ===

    // 设置设备容量上限并装载 [0, n) 全部 BlockId 到 freeList。
    // 必须在 Allocate / Release 之前调用一次。
    // 多次调用:同 n 幂等,不同 n 返回 INVALID_PARAMS。
    int32_t SetMaxBlockCount(uint64_t n);

    // 设置 TRIM 上下文(Open 阶段调,Close 时清理)
    void SetTrimContext(int dev_fd, bool trim_supported);

    // 设置可配置阈值(Open 阶段从 Options 透传)
    void SetTuning(double switch_ratio,
                   double reject_ratio,
                   double symmetric_ratio,
                   uint64_t min_recycle_threshold);

    // === 业务核心 ===

    // 分配一个 BlockId,严格升序(每次返回 freeList 当前最小者)。
    // 必须在 Engine.unique_lock 下调用(契约约束)。
    int32_t Allocate(BlockId* out);

    // 释放一个 BlockId,进 active recycle,等切换后才能再被分配。
    // 同步发 TRIM(ioctl(BLKDISCARD)),失败 WARN 不影响返回。
    int32_t Release(BlockId id);

    // 批量释放,原子语义(全成功或全失败,reserve 失败则整批不 push)。
    int32_t ReleaseBatch(const std::vector<BlockId>& ids);

    // === P5 WAL recovery 用 ===

    // 从 active block 集合反推 freeList(WAL replay 后调用一次)。
    // 行为:freeList = [0, max) - active(降序),active/inactive 清空。
    int32_t RebuildFromActive(std::span<const BlockId> active);

    // === 观察 / Stats(独立 getter)===

    uint64_t FreeCount() const;             // 全局可用(三者之和,兼容旧语义)
    uint64_t AllocatableCount() const;      // 立即可分配(freeList.size())
    uint64_t PendingRecycleCount() const;   // 等切换才可分配(active + inactive)
    uint64_t SwitchCount() const;            // 累计切换次数
    uint64_t MaxBlockCount() const;          // 设备总容量
    bool TrimSupported() const;
    bool IsSortInProgress() const;          // 当前是否处于 Switching 状态

private:
    // 三容器 + idx 管理
    std::vector<BlockId> containers_[3];
    uint8_t free_idx_ = 0;
    uint8_t active_idx_ = 1;
    uint8_t inactive_idx_ = 2;

    // 容量与配置
    uint64_t max_block_count_ = 0;
    uint64_t initial_capacity_ = 0;
    double switch_ratio_ = 0.90;
    double reject_ratio_ = 0.90;
    double symmetric_ratio_ = 1.5;
    uint64_t min_recycle_threshold_ = 1024;

    // 状态机 + 计数
    enum class State : uint8_t { Running, Switching };
    std::atomic<State> state_{State::Running};
    std::atomic<bool> sort_done_{false};
    std::atomic<bool> switch_pending_{false};
    std::atomic<uint64_t> switch_count_{0};

    // 后台 sort worker
    std::thread sort_worker_;
    std::mutex worker_mu_;
    std::condition_variable worker_cv_;
    std::vector<BlockId> sort_task_;
    std::vector<BlockId> sort_result_;
    std::atomic<bool> stop_{false};
    bool initialized_ = false;

    // TRIM 上下文
    int dev_fd_ = -1;
    bool trim_supported_ = false;

    // 内部
    void StartSwitch();
    void CompleteSwitch();
    void SortWorkerFn();
    void IssueTrim(BlockId id) const;
    bool ShouldTriggerSwitch() const;
};
```

### 6.2 调用契约

**业务线程契约**:
- `Allocate` / `Release` / `ReleaseBatch` 必须在 Engine.unique_lock 下调用
- `RebuildFromActive` 必须在 Engine.unique_lock 下调用,且只在 recovery 阶段(无业务并发)
- `SetMaxBlockCount` / `SetTrimContext` / `SetTuning` 必须在 Open 阶段调用一次,在 Allocate / Release 之前
- 观察接口(`FreeCount` / Stats 等)可在 shared_lock 下调用

**Engine 与 FreeList 的协作**:
- `Engine::Open` 调:
    - `freeList_.SetMaxBlockCount(io_.BlockCount())`
    - `freeList_.SetTrimContext(io_.GetDeviceFd(), probe_result)`
    - `freeList_.SetTuning(opts.freelist_switch_ratio, opts.freelist_reject_ratio, opts.freelist_symmetric_ratio, opts.freelist_min_recycle_threshold)`
- Engine 析构时由 FreeList 自身析构停止后台线程

## § 7 核心算法

### 7.1 启动初始化(`SetMaxBlockCount`)

```cpp
int32_t SetMaxBlockCount(uint64_t n) {
    if (initialized_ && n != max_block_count_) {
        return INVALID_PARAMS;  // 不同 n 拒绝
    }
    if (initialized_) {
        return SUCCESS;  // 同 n 幂等
    }

    max_block_count_ = n;

    // 装载 [0, n) 到 containers_[0],降序(便于 pop_back 取最小)
    containers_[0].clear();
    try {
        containers_[0].reserve(n);
    } catch (...) {
        return MEMORY_INSERT_FAIL;
    }
    for (uint64_t i = n; i > 0; --i) {
        containers_[0].push_back(i - 1);  // n-1, n-2, ..., 1, 0
    }
    containers_[1].clear();
    containers_[2].clear();

    free_idx_ = 0;
    active_idx_ = 1;
    inactive_idx_ = 2;
    initial_capacity_ = n;

    // 启动后台 sort worker
    stop_.store(false);
    sort_worker_ = std::thread([this] { SortWorkerFn(); });

    initialized_ = true;

    CABE_LOG_INFO("FreeList initialized: max_block_count=%lu, "
                  "switch_ratio=%.2f, reject_ratio=%.2f, "
                  "symmetric_ratio=%.2f, min_recycle_threshold=%lu",
                  n, switch_ratio_, reject_ratio_,
                  symmetric_ratio_, min_recycle_threshold_);
    return SUCCESS;
}
```

40T 设备:`reserve(41M)` + `push_back × 41M` ≈ ~300 ms,可接受。

### 7.2 Allocate

```cpp
int32_t Allocate(BlockId* out) {
    if (out == nullptr) return MEMORY_NULL_POINTER_EXCEPTION;
    if (!initialized_) return DEVICE_NO_SPACE;

    // Step 1: 若 sort 已完成,完成切换(O(1))
    if (state_.load(std::memory_order_acquire) == State::Switching
        && sort_done_.load(std::memory_order_acquire)) {
        CompleteSwitch();
    }

    // Step 2: 写保护(全局可用 ≤ 10% 拒绝)
    uint64_t available = containers_[free_idx_].size()
                       + containers_[active_idx_].size()
                       + containers_[inactive_idx_].size();
    if (available <= static_cast<uint64_t>(
            max_block_count_ * (1.0 - reject_ratio_))) {
        return DEVICE_NO_SPACE;
    }

    // Step 3: 从 freeList 取(降序 vector,pop_back 取最小)
    if (containers_[free_idx_].empty()) {
        // freeList 暂时空(切换中且 sort 未完成),业务上层重试
        return DEVICE_NO_SPACE;
    }
    *out = containers_[free_idx_].back();
    containers_[free_idx_].pop_back();

    // Step 4: 检测切换触发
    if (state_.load(std::memory_order_acquire) == State::Running
        && ShouldTriggerSwitch()) {
        StartSwitch();
    }

    return SUCCESS;
}

bool ShouldTriggerSwitch() const {
    // 必要条件:active recycle 达到最小阈值(R-NEW-1)
    if (containers_[active_idx_].size() < min_recycle_threshold_) {
        return false;
    }
    // 充分条件:freeList 已用 90% 或对称水位
    bool free_threshold = containers_[free_idx_].size()
        <= static_cast<uint64_t>(initial_capacity_ * (1.0 - switch_ratio_));
    bool symmetric = containers_[active_idx_].size()
        >= static_cast<uint64_t>(containers_[free_idx_].size() * symmetric_ratio_);
    return free_threshold || symmetric
        || switch_pending_.load(std::memory_order_acquire);
}
```

### 7.3 Release / ReleaseBatch

```cpp
int32_t Release(BlockId id) {
    if (!initialized_) return POOL_NOT_INITIALIZED;

    containers_[active_idx_].push_back(id);

    // 同步发 TRIM(失败 WARN)
    IssueTrim(id);

    // 标记对称水位 pending(实际切换在 Allocate 完成)
    if (state_.load(std::memory_order_acquire) == State::Running) {
        if (containers_[active_idx_].size()
                >= containers_[free_idx_].size() * symmetric_ratio_
            && containers_[active_idx_].size() >= min_recycle_threshold_) {
            switch_pending_.store(true, std::memory_order_release);
        }
    }

    return SUCCESS;
}

int32_t ReleaseBatch(const std::vector<BlockId>& ids) {
    if (!initialized_) return POOL_NOT_INITIALIZED;
    if (ids.empty()) return SUCCESS;

    // reserve 一次,避免多次扩容(原子性:reserve 失败则整批不 push)
    try {
        containers_[active_idx_].reserve(
            containers_[active_idx_].size() + ids.size());
    } catch (...) {
        return MEMORY_INSERT_FAIL;
    }

    // reserve 成功后 push_back noexcept,整批成功
    for (BlockId id : ids) {
        containers_[active_idx_].push_back(id);
        IssueTrim(id);
    }

    // 检测对称水位(同 Release)
    if (state_.load(std::memory_order_acquire) == State::Running) {
        if (containers_[active_idx_].size()
                >= containers_[free_idx_].size() * symmetric_ratio_
            && containers_[active_idx_].size() >= min_recycle_threshold_) {
            switch_pending_.store(true, std::memory_order_release);
        }
    }

    return SUCCESS;
}
```

### 7.4 StartSwitch

```cpp
void StartSwitch() {
    // 持 Engine 锁内调用,所以 active/inactive 索引访问安全
    {
        std::lock_guard lk(worker_mu_);
        sort_task_ = std::move(containers_[active_idx_]);
        containers_[active_idx_].clear();
    }

    // swap active 与 inactive 角色:
    //   旧 active 容器(刚 move 走,已空)→ 变 inactive
    //   旧 inactive 容器(可能含上次残余)→ 变 active,继续接收新 Release
    std::swap(active_idx_, inactive_idx_);

    state_.store(State::Switching, std::memory_order_release);
    sort_done_.store(false, std::memory_order_release);
    switch_pending_.store(false, std::memory_order_release);

    worker_cv_.notify_one();
}
```

总耗时:< 1 μs(几次 atomic store + 一次 vector move + 一次 idx swap)。

### 7.5 CompleteSwitch

```cpp
void CompleteSwitch() {
    {
        std::lock_guard lk(worker_mu_);
        containers_[inactive_idx_] = std::move(sort_result_);
        sort_result_.clear();
        sort_done_.store(false, std::memory_order_release);
    }

    // swap free 与 inactive 角色:
    //   旧 free 容器(含残余)→ 变 inactive,等下一轮 swap 进 active
    //   旧 inactive 容器(已 sort 完)→ 变 freeList
    std::swap(free_idx_, inactive_idx_);

    initial_capacity_ = containers_[free_idx_].size();
    state_.store(State::Running, std::memory_order_release);
    switch_count_.fetch_add(1, std::memory_order_relaxed);
}
```

总耗时:< 1 μs。

### 7.6 后台 sort worker

```cpp
void SortWorkerFn() {
    while (true) {
        std::vector<BlockId> task;
        {
            std::unique_lock lk(worker_mu_);
            worker_cv_.wait(lk, [this] {
                return stop_.load(std::memory_order_acquire)
                    || !sort_task_.empty();
            });
            if (stop_.load(std::memory_order_acquire)) return;
            task = std::move(sort_task_);
            sort_task_.clear();
        }

        // 锁外执行 sort(几百 ms,业务零阻塞)
        std::sort(task.begin(), task.end(), std::greater<BlockId>());

        // 回填 sort_result_,通知业务线程
        {
            std::lock_guard lk(worker_mu_);
            sort_result_ = std::move(task);
            sort_done_.store(true, std::memory_order_release);
        }
    }
}
```

### 7.7 RebuildFromActive(P5 用)

```cpp
int32_t RebuildFromActive(std::span<const BlockId> active) {
    if (!initialized_) return POOL_NOT_INITIALIZED;

    // 等正在进行的 sort 完成(recovery 阶段不应该有 sort,但保险)
    if (state_.load() == State::Switching) {
        while (!sort_done_.load(std::memory_order_acquire)) {
            std::this_thread::yield();
        }
        CompleteSwitch();
    }

    // 排序 active(便于求补集)
    std::vector<BlockId> sorted_active(active.begin(), active.end());
    std::sort(sorted_active.begin(), sorted_active.end());

    // 重置三容器
    for (auto& c : containers_) c.clear();

    try {
        containers_[0].reserve(max_block_count_ - sorted_active.size());
    } catch (...) {
        return MEMORY_INSERT_FAIL;
    }

    // 构造 freeList = [0, max) - active,降序排列
    auto rit = sorted_active.rbegin();
    for (uint64_t i = max_block_count_; i > 0; --i) {
        BlockId b = i - 1;
        if (rit != sorted_active.rend() && *rit == b) {
            ++rit;
            continue;
        }
        containers_[0].push_back(b);
    }

    free_idx_ = 0;
    active_idx_ = 1;
    inactive_idx_ = 2;
    initial_capacity_ = containers_[0].size();
    switch_count_.store(0);

    CABE_LOG_INFO("FreeList rebuilt from %zu active blocks, "
                  "freeList size=%zu",
                  active.size(), containers_[0].size());
    return SUCCESS;
}
```

### 8.2 Release 路径同步发 TRIM

```cpp
void FreeList::IssueTrim(BlockId id) const {
    if (!trim_supported_ || dev_fd_ < 0) return;

    uint64_t range[2] = {
        id * CABE_VALUE_DATA_SIZE,    // offset
        CABE_VALUE_DATA_SIZE           // length(1 MiB)
    };

    int rc = ::ioctl(dev_fd_, BLKDISCARD, range);
    if (rc != 0) {
        CABE_LOG_WARN("BLKDISCARD failed for blockId=%lu: errno=%d",
                      id, errno);
        // 不返回错误,TRIM 失败不影响正确性
    }
}
```

性能:
- 单次 `ioctl(BLKDISCARD)` 1 MiB 范围:~10-100 μs
- Release 频率(cabe 主场景):远低于 Put,影响可忽略
- 大对象 Delete 路径整体延迟 ~1 ms 量级,TRIM ~10-100 μs 可接受

## § 9 错误处理与边界

### 9.1 边界情况清单

| 边界 | 处理 |
|---|---|
| 启动后纯写入,freeList 接近耗尽但 `active < min_recycle_threshold` | 不触发切换,继续消耗 freeList。极端时撞写保护拒绝 |
| freeList 完全空但切换未完成 | Allocate 返回 `DEVICE_NO_SPACE`,业务上层重试(等 sort 完成) |
| `ReleaseBatch` 中 `reserve` 失败 | 整批不 push,返回 `MEMORY_INSERT_FAIL`,recycle 状态不变 |
| `RebuildFromActive` 时 active 集合包含 ≥ max 的 BlockId | 忽略越界 BlockId(防御性),记 WARN |
| `Engine.Close` 时切换中 | FreeList 析构等 sort worker join,丢弃 sort_result(P4.5 不持久化) |
| TRIM 失败 | WARN 日志,不影响 Release 返回 |
| TRIM 不支持的设备(loop device 默认) | Open 阶段检测,后续跳过所有 TRIM 调用 |

### 9.2 不在 P4.5 处理的边界

- **多业务线程并发**:依赖 Engine 大锁,P6 reactor 阶段再优化
- **跨多 NVMe 设备**:FreeList 单设备语义,多盘由 Engine 上层 BlockId→DeviceIndex 策略处理
- **后台 sort 异常**:`std::sort` on `uint64_t` 是 noexcept,不会抛异常

## § 13 性能特征

### 13.1 操作复杂度

| 操作 | 复杂度 | 实测耗时(40T 设备) |
|---|---|---|
| Allocate(命中) | O(1) | ~20-50 ns(vector pop_back + 阈值检查) |
| Allocate(触发 StartSwitch) | O(1) | ~1 μs(swap idx + atomic) |
| Allocate(触发 CompleteSwitch) | O(1) | ~1 μs |
| Release | O(1) | ~50 ns + TRIM 10-100 μs |
| `ReleaseBatch(N)` | O(N) | ~10 ns × N + TRIM N 次 |
| StartSwitch | O(1) | < 1 μs |
| CompleteSwitch | O(1) | < 1 μs |
| Sort(后台) | O(N log N),N = active.size | 0.5-2 秒(异步,不阻塞) |
| `RebuildFromActive` | O(max + active log active) | 0.5-1 秒(Open 阶段调用,可接受) |

### 13.2 切换频率与持锁时间

40T 设备稳态(switch_ratio = 0.90):
- 切换间隔:30 分钟 - 数小时
- 单次切换持 Engine 锁总时间:~2 μs(StartSwitch + CompleteSwitch)
- 系统锁占用率:0.0000001%

**业务路径几乎完全无阻塞**。

### 13.3 SSD 友好性

- 顺序分配阶段(启动到第一次切换):完全 LBA 升序写入,SSD WAF ≈ 1
- 切换后:freeList 内容由前一轮 active sort 而来,分配仍升序,但跨切换会有 LBA 跳跃
- 长期总体:接近顺序,WAF < 1.1

### 13.4 内存占用

40T 设备:
- 三容器总和稳态约 = 全局可用 BlockId 数 × 8 字节
- 40T 设备 = 41M × 8 字节 = **328 MiB**(分散在三容器)
- 加上 sort_task / sort_result 临时缓冲:~100-200 MiB 峰值
- 总:~500 MiB

可接受。

## § 16 与 P5 WAL 衔接

### 16.1 P5 recovery 流程对接

P5 WAL recovery 完成后,`Engine::Open` 在 WAL replay 之后调:


`RebuildFromActive` 已在 P4.5 实现(D-13),P5 直接调用。

### 16.2 freeList 不需要 WAL 帧

freeList 状态完全可从 chunkIndex 推导(R-NEW-4 已分析)。WAL 帧只需记录 metaIndex / chunkIndex 变更,不需要记录 freeList 切换 / Release / Allocate。

这简化了 P5 WAL 帧设计。

### 16.3 切换与 WAL 的解耦

P4.5 阶段 freeList 切换是纯内存行为,**与 WAL 完全无关**,在 P5 阶段也保持解耦。切换频率不影响 WAL 大小,WAL truncation 不影响切换语义。

## 附录 C:术语表

| 术语 | 含义 |
|---|---|
| freeList | 当前作为分配源的容器(`containers_[free_idx_]`),降序存储 |
| active recycle | 当前接收 Release 的容器(`containers_[active_idx_]`),无序 |
| inactive recycle | 当前不接收也不分配的容器(`containers_[inactive_idx_]`),可能为空或含残余 |
| 轮换 / 切换 | 通过 idx swap 实现容器角色变换,无数据移动 |
| 残余 | CompleteSwitch 后旧 freeList 中未消费的 BlockId,留在 inactive 容器 |
| StartSwitch | 切换的起始动作,把 active 送 sort + idx swap |
| CompleteSwitch | 切换的完成动作,sort 完成后 idx swap |
| `min_recycle_threshold` | 切换前 active 必须达到的最小 BlockId 数(默认 1024) |
